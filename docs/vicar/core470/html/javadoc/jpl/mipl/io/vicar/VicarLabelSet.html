jpl.mipl.io.vicar

Class VicarLabelSet

  • java.lang.Object
    • jpl.mipl.io.vicar.VicarLabelSet
  • All Implemented Interfaces:
    java.io.Serializable, java.lang.Cloneable, java.lang.Iterable, java.util.Collection, java.util.Set

    public class VicarLabelSet
extends java.lang.Object
implements java.lang.Cloneable, java.io.Serializable, java.util.Set
    This class maintains an entire set (property set, history task, or system) of VICAR labels, with a collection of key/value pairs (Item's).

    Note that by definition of VICAR labels, the order of items within a set is irrelevant and unspecified, and duplicates are not allowed. This class attempts to maintain the order in which the items are provided to it, but this is in no way guaranteed. (This is the same behavior as the C RTL).

    Access to this object is not synchronized in any way. The caller must synchronize if necessary. ArrayLists are used instead of Vectors for efficiency.

    This class ensures that History and Property sets (as defined by the set's "flavor") start with the appropriate keywords ("PROPERTY" for property sets, "TASK", "USER", and "DAT_TIM" for history tasks). The values of these can be modified but the items themselves cannot be removed or unset. Note that per VICAR conventions, the property and task names are converted to upper case.

    The set's "flavor" (system, property, or history) must be set at construction time and cannot be changed (due to the required header labels).

    This class implements Set so it can be used as a collection. Additional constraints imposed beyond those of Set: all elements must be instances of VicarLabelItem.

      • add

        public boolean add(VicarLabelItem new_item)
        Add an item to the list. If the given keyword matches one already in the list, the old item is replaced. Otherwise, the item is added to the end. The reserved names LBLSIZE, PROPERTY, TASK, USER, and DAT_TIM may only be added to the proper flavor of label (otherwise an IllegalArgumentException will be thrown).

        If PROPERTY or TASK is set, the value is converted to uppercase, per VICAR convention.

        Note: This function violates the contract of Set.add in the following way: if the item already exists (according to the keyword), the new item replaces the old one. This modification is needed because item equality is determined only by the keyword; the value is not checked. Thus it is useful to replace the existing item, as opposed to a true Set where equality is defined by the entire object.

        Returns:
        true always
        Throws:
        java.lang.IllegalArgumentException

      • toLabelString

        public int toLabelString(java.lang.StringBuffer buf,
                         int max_length,
                         int start_index)
        Returns in the supplied StringBuffer the part of the set from start_index to the end as a valid VICAR label string, ready to be put in a file. If the max_length argument is non-zero, it represents the maximum size the string can become (normally used to make sure the label fits in the space at the front of a file). This includes any data already in buf when the function is called. A 0 means no maximum.

        Note that the caller is responsible for putting in LBLSIZE= at the beginning of each buffer's label segment.

        Returns:
        The first index not put in the string (due to overfilling the capacity); i.e. it is the index you would supply to start_index in the next call. Returns -1 if all items fit.

      • fromXML

        public void fromXML(org.w3c.dom.Element set,
                    java.util.List errorList)
        Reads the Set from an XML DOM node (either <system>, <property>, or <history>). Items will be created for each subnode as necessary. See the VICAR label DTD. !!!!TBD: reference to this DTD!!!!

        Items found in the XML are added to the set. This means that existing items will be retained in the output if not overridden. (the behavior of add() is to replace items with the same keyword). If you want only the XML items, make sure the Set is empty before calling this function.

        Note that the label "flavor" (system/property/history) cannot be changed by this routine. If the supplied DOM node does not match the already-set flavor, a warning is issued but the flavor is not changed. This means that the header items will very likely be wrong.

        If anything is wrong with the supplied node, error/warning messages will be appended to the supplied errorList. However, the parser keeps working as long as possible, despite errors. The practical upshot of this is that the errors can usually be ignored; the parser will do the best it can. However, the errors can be displayed to the user if desired. An exception will be thrown only if something unrecoverable happens.

        If the required header attributes are not present (TASK, DAT_TIM, etc.), the appropriate label items will be unchanged. This generally means they are set with the value "UNSET" (or "unset") per initItems(). Reasonable values for these should be supplied before writing the label. Note that warnings will be issued via errorList if the attributes are missing.

        Parameters:
        set - The Element from which to extract the item info.
        errorList - A list of Strings to which any parsing errors will be appended. Can be null, in which case no errors are returned.
        Throws:
        org.w3c.dom.DOMException - if something unrecoverable happens. Improperly formatted items or violations of the DTD do not necessarily result in exceptions; parsing continues as long as possible and problems are appended to the errorList.
        See Also:
        VicarLabelItem.fromXML(Element, List), initItems()
