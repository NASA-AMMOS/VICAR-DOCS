jpl.mipl.mica.pig

Class CoordSystem

  • java.lang.Object
    • jpl.mipl.mica.pig.CoordSystem

  • public class CoordSystem
extends java.lang.Object
    Defines a coordinate system. CS's are defined by three identifiers: Mission, Frame, and Site. Mission is MPF, M98, etc. and basically reflects the subclass. The mission defines what allowable Frames there are. Frame is the definition of a coordinate system, for example Surface Fixed, Local Level, Lander, MVACS (m98 only), etc. Site is a pointer to a (shared) object that defines mission-specific parameters for each unique location (if the coordinate system moves due to spacecraft or rover motion, that would be a different site).

    Each frame is referenced back to a single Fixed coordinate system for that mission. For MPF this is called Mars Fixed and for M98 it is Surface Fixed. This Fixed coordinate system has the property that features on the body of interest do not move. For orbiters this might be a body-center-fixed CS which is easily translated into lat/lon. The Fixed frame is used to transform between various other frames, which may vary over time (different sites). Note that orbiters might very well have a new site for each image (or even every line for a pushbroom scanner).

    Transformation between missions is not implemented now, but could be done by relating the two base Fixed coordinate systems to each other using some uber-alles coordinate system (e.g. lat/lon of each landing site). Such parameters would most likely be in Config files, not in indvidual image labels, due to their global nature.

    Transformation between frames is normally done by transforming to the common Fixed frame, then back into the destination frame. This may be short-circuited in some common cases for efficiency (e.g. M98 MVACS <-> Lander transformations when the site is the same; it's just a sign change and offset). Note that even these would have to go through the Fixed frame if they are at different sites, because the relative orientations would no longer be the same.

    Mission and frame ID's are strings, but are converted internally to ints for ease of comparison. These ints are mission-dependent.

    CS objects, like sites, are shared as much as possible. The create functions search a static global table of CS objects, and return an existing object if the mission, frame, and site match. This ensures that items which should be in the same coordinate system, are all using the same parameters.

    CS subclasses must not cache data obtained from the Site object. The Site may change at any time (in the future, Sites will probably be attached to PointingModel's so they can be nav'd as well as individual image pointing). It is up to the mission-specific Site subclass to compute and cache any commonly used items whenever the Site parameters change.

    CS object support conversions of three quantities: Point, Vector, and az/el. Vectors are simply rotated as necessar; there is no origin (they are just an orientation, typically they are unit vectors but this is not required). Points are rotated as well, but are translated too based on the offsets between the CS's (they are a vector from the CS origin to the given physical point, or equivalently, cooridnates of a point in the CS). Az/el angles are simply a convenience to avoid having to go to/from a vector all the time.

    Note that EVERY time az/el is converted to/from an orientation vector, the corresponding CS routines MUST BE USED! The reason is that most CS's define positive elevation as toward the +Z axis (e.g. M98 Fixed, LL, MVACS) while others define it as toward the -Z axis (e.g. MPF all, M98 Lander) (this mostly happens when +Z points down). The CS objects take this into account. Plus, is it possible for some CS's to use the equivalent of West longitude while others use East.

    Note that az/el are always measured in radians.

    NOTE: This object is a proxy object for a C++ implementation. The C++ implementation is part of the NASA/JPL Planetary Image Geometry (PIG) library. This object holds a pointer to an underlying C++ PIG object and makes its methods available to Java. For information on how to create and use PIG objects in Java, see the jpl.mipl.mica.pig package information. For more information on the PIG library and other related JPL software, see MIPS.

    This proxy is different from other mica proxy objects. The under lying c++ impl is difficult to work with. To make this work with Surface models the initialize method DOES NOT stuff a c++ address into the java variable. If you create a surface model and call SurfaceModel.getCoordSystem you will get a valid (both java and c++ are init'ed) proxy.
