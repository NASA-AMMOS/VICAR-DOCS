Class IsisEDRReader

  • java.lang.Object
    • jpl.mipl.tlm.jedi2.IsisEDRReader
  • All Implemented Interfaces:
    EDRReader

    public class IsisEDRReader
extends java.lang.Object
implements EDRReader
    An EDR Object assembler that ingests a Cassini ISS VIMS cube (for use in the JEDI Display Manager/Server). Basically, to use this object, you simply instantiate it with a ISIS file as input, stand back, and hope all goes well. You probably should not call this class directly. Use the EDRLoader class instead (it implements the EDRReader interface as well).

    Also, this object should handle EDR sized images fairly well. I can make no claims as to what will happen if you feed it a huge dataset.

    NOTE: This implementation is really designed to read the Cassini VIMS EDRs as generated by Raj's Instrument Constructor.

      • IsisEDRReader

        public IsisEDRReader(java.lang.String infile,
                     Personality pers)
        IsisEDRReader create an EDR object populated with the meta-data and three planes worth of the instrument data as read from the Cube. The int[] allows for the selection of which three planes will be used, in r, g, b order.

        Note: Currently, the planes are numbered starting at 0, not 1.

      • IsisEDRReader

        public IsisEDRReader(java.lang.String infile)
        IsisEDRReader will only place three planes (bands) of the Cube data into the EDR object (since that's all we can really see in the current widgets). This constructor simply calls the full constructor with the selection of the first three bands for the r, g, and b respectively.

        loadFile will ingest the entire contents of the ISIS label and the image data. Since I'm assuming that RealTime images *tend* to be on the order of 1k x 1k, this shouldn't be a problem. I don't know what I'll do when EDRs grow to be 100 Mbytes each ;-) --MISR for example--. But, one step at a time.

        loadFile does not currently support binary headers or prefixes.
