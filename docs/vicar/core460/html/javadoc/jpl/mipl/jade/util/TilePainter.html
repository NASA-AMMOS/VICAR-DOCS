jpl.mipl.jade.util

Class TilePainter

  • java.lang.Object
    • jpl.mipl.jade.util.TilePainter
  • All Implemented Interfaces:
    java.util.EventListener, TileComputationListener

    public class TilePainter
extends java.lang.Object
implements TileComputationListener
    A class that manages painting of a single tile in an asynchronous environment. This class keeps track of whether a tile request is currently queued, as well as managing the repainting of the tile once an asynchronous request is completed.

    This class should only be used by JadeDisplay; it is not intended to be called from user code.

    Author:
    Bob Deen, JPL

      • paintTile

        public boolean paintTile(TileCache cache,
                java.awt.image.RenderedImage rendering,
                java.awt.Graphics g)
        This function is called when a tile needs to be repainted. The caller should have already checked to see if this tile needs to be repainted. The tile will be immediately gathered and painted, or enqueued for computation on a JAI worker thread, depending on the setting of repaintPolicy and whether or not the tile is in the cache.

        This function assumes that the caller has already updated the damage list based on whatever triggered this call. However, TilePainter is responsible for removing the tile from the damage list once it is actually painted to the screen.

        If repaintPolicy is anything other than REPAINT_DEFERRED (i.e. there's any chance of immediate painting), then this function must be called from the AWT/Swing event thread, in order to avoid confusing the single-threaded Swing.

        If a this is called from a paint method (or its cousins), then a Graphics object should be passed in. This object will be used only for immediate-mode painting. If it is null, or if painting is deferred, then JComponent.getGraphics() will be used instead to get a Graphics to use. This is unfortunately necessary due to Swing's double buffering scheme... if we paint from within a paint method, we could be double buffered whether we want to be or not (e.g. JViewport seems to always double-buffer, regardless of settings in the component or the RepaintManager). So, we have to paint in the supplied Graphics object. In deferred mode, we paint in an invokeLater method, which executes in the event thread and is by definition not inside any repaint method calls, so direct access to the JComponent.getGraphics() is fine. Yuck.

        Parameters:
        cache - the TileCache associated with the image. Used only for repaintPolicy == REPAINT_CACHE. null is acceptable and means no cache is available, so all tiles will be deferred.
        rendering - the RenderedImage used as a key to look up the tile in the cache. Renderings may change even though the image object itself doesn't.
        g - the Graphics object to use for immediate mode if we're in a paint method (or one of its relatives), or null otherwise.
        Returns:
        true if the tile was painted; false if it was deferred

      • paintTileNow

        protected void paintTileNow(java.awt.image.Raster tile,
                java.awt.Graphics g)
        Paint the tile immediately. We must already be in the AWT/Swing event thread. We use the supplied Graphics object if present, or get a new graphics (indirectly via JComponent.getGraphics()) if it is null. The Graphics object should be supplied if this is called from within a paint method in order to handle Swing double buffering.
