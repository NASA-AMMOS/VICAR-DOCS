    Plutch - PLaces Update Telemetry Common Header

    This class reads the common header (contained in a .emd file) that is attached to all data products, extracts the data needed for PLACES, and outputs an XML file suitable for submission to PLACES (typically done via "curl").

    The supplied filename should be that of the .emd file. Multiple data products can be processed at once (this is more efficient when doing the actual upload).

    This class is subtly different from "plumage" in that plumage looks at the image labels, while this looks at the .emd file. While those are typically the same for engineering cameras, they need not be the same for MMM or RMI. The common header is populated at the time of data product creation, while the image IDPH is populated at the time of image acquisition. These need not be the same.

    This class must be as bulletproof as possible, so it can be called from MATIS directly and not affect the MATIS JVM. Alternatively, a main() function exists, which can be used instead and in any case acts as a test harness.

    Bob Deen, JPL

        public static int process(java.util.List<java.lang.String> inp,
          java.lang.String out,
          java.lang.String min_sclk_str,
          java.lang.String max_sclk_str,
          int add_site,
          java.io.PrintStream outStream)
        This function implements the functionality of Plutch. Inputs are a list of filenames (.emd files), and an output filename to be written to. Unlike the main() driver, the output is not tested and will be overwritten if it already exists. If the minSclk or maxSclk parameters are non-null, they will be compared against the actual sclk. If the sclk is outside the range, the file is ignored. The add_site parameter is an integer added all Site values. This is a testing aid, allowing one to test PLACES upload using areas of the dev database that don't have old uploads in the way. It should NEVER be used in operations! The outStream parameter is a PrintStream object where all logging type messages will be written. Any products with a value other than 8 in msl-emd:FswMode are ignored. State 8 means the product was created by an RCE in prime state, which is what we want. 17 means it was "online" (basically the backup processor), in which case the rover state is almost certainly wrong. 0 means the FSW state is unknown, which makes the rover state highly suspect. Other values should not occur but are also ignored for safety. Note that if a product is created by RCE A while prime and stored, and then later we switch to B as prime and downlink the product, it shows up as a "cross-string" product because it comes from the currently non-prime RCE's memory. But the product is good because A was prime when the product was created, thus the rover state was good at that time. This is reflected as an FSW state of 8.
        Returns:
        0 on success
