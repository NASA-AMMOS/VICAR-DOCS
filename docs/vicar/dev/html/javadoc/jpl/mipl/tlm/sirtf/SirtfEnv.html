Class SirtfEnv

  • java.lang.Object
    • jpl.mipl.tlm.sirtf.SirtfEnv

  • public class SirtfEnv
extends java.lang.Object
    Encapsulates various environment resources that might be needed to process SIRTF telemetry. Such resources are:
    a validated UPF
    SIRTF specific checks that ensure the parameters set in the UPF are correct and appropriate for use. Many settings and dependencies are checked to ensure that the telemetry processing code can use more intuitive accessor methods.
    logging infrastructure
    a parsed CPT (Channel Parameter Table)
    SIRTF telemetry processinginvolves creating FITS files for the science instrument data. FIS are vaguely similar to VICAR in that the header is in standard ASCII encoding that denotes keyword/value entries. The project CPT is used to help populate the FITS header with channel designations and comments. SirtfEnv will parse the CPT specified in the UPF and make the contents available in TreeMap form.
    a database connection
    This telemetry processing application uses XML persistence for a handful of critical JavaBeans. As such, the dependency on a database are greatly reduced. This is not to say that a database is unnecessary, though. The database is, ultimately, the final repository for runtime progress and archival QQC information. Also, SQL generated reports are of use. SirtfEnv, therefore, creates and provides access to a database connection (JDBC).
    an FEI manager Thread, if needed
    Currently, FEI is accessed through a command-line API. If FEI is enabled for processing, a manager Thread will be created to receive requests for adding files to FEI fileTypes.

      • isIracScalingEnabled

        public boolean isIracScalingEnabled()
        States whether or not the UPF has designated that unsigned scaling be applied to the IRAC science data contained within the file. The FITS standard only recognized short integers as signed. However, some science instruments have modes which create data in unsigned short format. In order to correctly place unsigned data into a FITS file, we must scale the range of (0,65535) to (-32768,32767) and note this in the FITS header. Operationally, MIPL is not required to perform this scaling, as SSC will be further modifying the files we give them anyway. During ATLO testing, when we directly interact with the instrument teams is the most common use of this setting.
        UPF Keyword: UNSIGNED_SCALING
        Default: OFF

      • isStalePacketProcessing

        public boolean isStalePacketProcessing()
        States whether or not packets that have an ERT older than 30 days from "now" are allowed to proceed through the processing stream. Under normal operational conditions, this should be false since the online repository of processed data is a rolling 30-day window. Should we ses a packet that, for whatever reason, got delayed to a point such that its ERT is older than 30 days, there is a real chance the we'd make a new product out of such packets and likely give the product a version number of 1. Clearly this isn't what we want as this stale data has probably been seen before and we've already processed it correctly. We would have known we had seen it if the packet had been younger but, in the face of such a packet, the best thing to do is to note it for the analyst but drop it on the floor for further processing.
        Set this keyword to "ON" should you wish to reprocess some old data.
        UPF Keyword: STALE_PACKET_PROCESSING
        Default: OFF
        Returns:
        true if stale packets should be processed
