      • collate

        public void collate(SirtfSFDU sfdu)
        Assigns an SFDU to a particular SirtfOutputProduct. collate() uses an internal algorithm to construct a unique 
        fileKey
        for each packet received. This fileKey is then used as a search key to find the associated destination SirtfOutputProduct. The search steps are as follows:
        1. is fileKey in our blacklist? If so, then the product is in a quizzical state and we should ignore the packet and move on. We will have to come back to this product and process it manually.
        2. is fileKey in our completed list? If so, then the product is already complete and we can ignore this, apparently duplicate, packet and move on.
        3. is fileKey in our partial Map? If so, then this is a partial product that we already know about in memory, and we can immediately proceed with this packet. If not, then
        4. is fileKey in out persisted partial List? If so, then we've persisted this product to disk to help free up resources for processing, but still have knowledge of the product in memeory. Reconstruct the SirtfOutputProduct from the persisted SirtfMetaData XML bean and data caches. If not, then
        5. is fileKey known within our local persisted store? If so, then we've processed the product before but, for various reasons, we don't have any knowledge about in memory (e.g., because we restarted the tlmproc) or in the DB (because it hasn't been updated yet). Reconstruct the product from the xml file and data caches. Searching for the correct XML is a little involved as we have temporal directories for the data. We may have to look through several, well-formed, directories to determine whether or not we've processed this file before. This is why the database is consulted ahead of this step. If no XML persistence file is found, then
        6. this is a new product. Create new objects for processing and begin constructing the product.

        One thing to remember: collate() is intended to only receive SFDU's that are valid. Blatantly invalid SFDU's never reach this method. However, there have been, on occasion, SFDU's that aren't invalid (in the sense used here), but are odd. The packets that exhibit such errant behavior can still cause an OutputProduct to be generated but those products will have their valid property set to false.

      • finish

        public boolean finish(int type)
        Will close out all remaining partial files (and persisted partial files). Additionally, the expedited and nominal processing reports will be generated. This method will return true if there was something to actually do, otherwise, false. False return implies that finish() was called while SirtfPacketStore had never seen any packets to deal with (such packets didn't make it through the SirtfTlmProc gate, as it were).

        There are four groupings of DCE's that can be flushed and closed via this method:

        • DCE_TYPE_ANY
        • DCE_TYPE_EXPEDITED
        • DCE_TYPE_NOMINAL
        • DCE_TYPE_BACKPRESSURE
